# Vault — Instructions for Claude Code

## What is this vault
Centralized long-term **memory layer** for all projects — persistent judgment
(decisions, why, status) across sessions. Notes follow the Zettelkasten method:
atomic (one idea per note), densely interlinked, standardized metadata.

## Structure — what is / isn't graphed
The memory graph's **scan root is `~/vault/memory/` only**; graphify ingests just
the distilled notes there. Everything else is raw/tooling, kept out of the graph
by structure (outside the scan root) or by `.graphifyignore`.

```
memory/
├── graphify-out/              GLOBAL tier: code-free, composed from memory graphs  ✗ auto-excluded
├── global/                    cross-project principles, via /ingest-principles ✓ graphed (global tier)
└── projects/<project>/
    ├── decisions/             ADRs, why-X-over-Y                   ✓ graphed → bridged to code
    ├── notes/                 permanent / concept notes           ✓ graphed → bridged to code
    ├── sessions/              summary-<chat-basename>.md: processed marker + recap ✓ graphed
    ├── chats/                 raw session transcripts              ✗ excluded
    └── graphify-out/          graph.json = MEMORY graph (notes only, input)
                               merged.json = MERGED query surface (repo ⊕ memory ⊕ bridges)
                               wiki/index.md = cached briefing      ✗ auto-excluded
```

Rule of thumb: **distilled judgment → `decisions/`+`notes/` (graphed); raw
narrative, transcripts, tooling → excluded.** `/save-memory` writes the raw
session into `chats/`, then `/ingest-session` distills it into `decisions/`+
`notes/` and writes the mirrored `sessions/summary-<basename>.md` — a transcript
counts as processed iff that mirrored file exists (no other marker; raw is
never edited to record that it was processed).

**The graph pipeline** (index is always `<dir>/graphify-out/wiki/index.md`):
- **Memory graph** `projects/<project>/graphify-out/graph.json` — this project's
  distilled notes only, re-extracted cheaply on `/save-memory`. An *input*: it feeds
  the merged graph and the global tier.
- **Per-project merged graph** `projects/<project>/graphify-out/merged.json` — the
  repo's code graph ⊕ the memory graph ⊕ the bridges `link-doc` draws between them.
  The **single query surface** for the project (code *and* why in one graph).
  Machine-local (contains vault nodes) — *not* the committed repo graph. Re-composed
  by the sync process (cheap code re-base on change; re-link every `/save-memory`).
- **Global tier** `memory/graphify-out/` — **code-free by construction**: *composed*
  (`merge-graphs`) from every project's memory graph + `global/`. Rebuilt on demand
  for cross-project navigation. `merge-graphs` is a mechanical, no-LLM union — it
  gives breadth (everything, cheaply) but no synthesis (two projects independently
  landing on the same rule stay two disconnected nodes). `global/` itself is the
  curated, high-signal layer: populated by **`/ingest-principles`**, run on demand
  (not per save), which reads every project's `decisions/`+`notes/`, judges which
  are genuinely cross-project (not just where they were discovered), and writes one
  generalized note per principle citing every project that evidences it via
  `sources:` — **not** a move like repo-doc promotion; a project's own note stays,
  since multiple projects can independently instantiate the same principle.

The map of content is **generated by graphify** (`export wiki`), never hand-written;
the briefings (`index.md`, …) are cached answers to pre-defined queries ("state of
this project") — read them, don't re-run the query.

## Zettelkasten Rules

### Note creation
- Use wikilinks: [[note-name]] (not markdown links)
- Mandatory YAML frontmatter on every note
- Filenames in kebab-case: `auth-flow.md`, not `Auth Flow.md`
- 1 concept per permanent note (atomicity)
- Minimum 2 wikilinks per note (dense linking)

### Standard frontmatter
```
---
title: Note Name
tags: [project, topic]
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: active
type: permanent
---
```

### Never do
- Don't delete notes without asking
- Don't use markdown links for internal notes (use wikilinks)
- Don't create notes without frontmatter
- Don't change folder structure without documenting it

## Session Commands

`/save-memory` and `/load-memory` are **global skills** (in
`~/claude-code-memory-setup/skills/`, symlinked into `~/.claude/skills/`), so they
work from any project directory — not only when this vault is the cwd. They read
the Zettelkasten rules **above** as the single source of truth when writing here;
this file defines *how* the vault is written, the skills decide *what* and
*where*. Do not re-add command logic to this file.

## Session transcripts (chats) and their distillation (sessions)
- `/save-memory` writes the raw session into `memory/projects/<project>/chats/`
  (frontmatter `type: chat`, tag `chat-import`) — the ground-truth record beneath
  the distilled notes. Excluded from the memory graph by the **per-project**
  `.graphifyignore` (created by `/save-memory` at the scan root — graphify's
  ignore walk stops at the nearest VCS root, so the vault-level file is NOT
  consulted for per-project scans unless `~/vault` itself is a git repo).
- **`/ingest-session`** (invoked by `/save-memory` right after the transcript is
  written, or standalone to backfill old ones) reads a `chats/<basename>.md`
  file — never this-conversation's own memory of itself — and writes
  `decisions/`/`notes/` entries plus a mirrored `sessions/summary-<basename>.md`.
  A transcript is "processed" iff that mirrored file exists — the analogue of
  a raw→summary link elsewhere in the design, adapted since one transcript maps
  to *many* atomic notes rather than one summary.
- Legacy imports from the old pipeline may still sit in top-level `chats/`.
- Graph View filters: `tag:chat-import` → chats only; `-path:chats` → hide chats.

## Graphify (memory graph tiers)
- **`memory/projects/<project>/graphify-out/`** — holds two artifacts: `graph.json`
  (the notes-only **memory graph**, an input) and `merged.json` (the **merged query
  surface**: repo code graph ⊕ memory graph ⊕ `link-doc` bridges — query with
  `--graph <that>/merged.json`). Machine-local, not committed. Kept fresh by the sync
  process (see the global config): cheap code re-base when code changes, re-link on
  `/save-memory`.
- **`memory/graphify-out/`** — the **global tier**, code-free, *composed* from the
  memory graphs + `global/` on demand for cross-project queries. `global/` itself is
  written by `/ingest-principles`, not by hand.
